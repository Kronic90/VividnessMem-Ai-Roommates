VividnessMem — How To Use (v1.0.6)
====================================

A standalone organic memory system for any LLM agent.
Zero dependencies beyond Python 3.10+ standard library.
(Optional: `pip install cryptography` for encrypted storage.)


INSTALL
-------
From PyPI:

    pip install vividnessmem

Or copy VividnessMem.py into your project. That's it.


QUICK START
-----------

    from VividnessMem import VividnessMem

    # Initialize — memories are saved to ./memory_data/ by default
    mem = VividnessMem()

    # Or pick your own directory:
    mem = VividnessMem(data_dir="my_agent/memories")

    # Professional mode — disables emotion/mood biasing for task agents
    mem = VividnessMem(professional=True)


ADDING MEMORIES
---------------
After your agent has a conversation, ask your LLM to decide what to
remember (use the built-in curation prompt), then store the results:

    from VividnessMem import VividnessMem, CURATION_PROMPT, parse_curation_response

    mem = VividnessMem()

    # 1. Send CURATION_PROMPT to your LLM at the end of a conversation
    llm_response = your_llm.generate(CURATION_PROMPT)

    # 2. Parse the JSON response
    entries = parse_curation_response(llm_response)

    # 3. Store each memory
    for entry in entries:
        if entry.get("bank") == "self":
            mem.add_self_reflection(
                content=entry["content"],
                emotion=entry.get("emotion", "neutral"),
                importance=entry.get("importance", 5),
                why_saved=entry.get("why_saved", ""),
            )
        elif entry.get("bank") == "social":
            mem.add_social_impression(
                entity="Alice",   # whoever the agent was talking to
                content=entry["content"],
                emotion=entry.get("emotion", "neutral"),
                importance=entry.get("importance", 5),
                why_saved=entry.get("why_saved", ""),
            )

    # 4. Save to disk
    mem.save()

Or add memories directly without the LLM curation step:

    mem.add_self_reflection(
        content="I really enjoy helping people debug code",
        emotion="happy",
        importance=7,
    )
    mem.save()


RETRIEVING MEMORIES (injecting into system prompt)
--------------------------------------------------
Every turn, call get_context_block() and inject the result into your
agent's system prompt:

    # Get the memory block for this turn
    context_block = mem.get_context_block(
        current_entity="Alice",              # who we're talking to
        conversation_context="last 2-3 messages here",  # for fg/bg split
    )

    # Inject it into your system prompt
    system_prompt = f"""You are a helpful assistant.

    {context_block}

    Respond naturally based on what you remember."""

Context-aware retrieval: pass context to get_active_self() to rank
memories by both vividness and topical relevance:

    relevant = mem.get_active_self(context="tell me about cooking")

Foreground/background split — saves prompt tokens while preserving
awareness of less-relevant memories:

    fg, bg = mem.partition_active_self(context="recent conversation text")
    # fg = full text for relevant memories
    # bg = compressed one-liners for the rest


RESONANCE (old memories resurfacing)
------------------------------------
Before building the context block, check for resonant memories:

    resonant = mem.resonate(context="the last few messages of conversation")

    context_block = mem.get_context_block(
        current_entity="Alice",
        resonant=resonant,
        conversation_context="the last few messages",
    )

Resonance finds old faded memories that match the current conversation
topic. It uses an inverted keyword index with bigram/trigram matching
for phrase-level precision, plus co-occurrence expansion.

Optional LLM semantic bridging for sparse results:

    resonant = mem.resonate(
        context="those old jazz records",
        llm_fn=your_llm.generate,  # used only when keyword results are thin
    )


MOOD SYSTEM
-----------
The mood system shifts the agent's emotional state gradually based on
conversation. It biases which memories surface (mood-congruent recall).
(Disabled in professional mode — mood stays neutral.)

    # After each exchange, update mood with emotions from the conversation
    mem.update_mood_from_conversation(["happy", "curious"])

    # Check current mood
    print(mem.mood_label)   # e.g. "happy"
    print(mem.mood)         # PAD vector, e.g. (0.6, 0.3, 0.2)


SHORT-TERM MEMORY (STM)
------------------------
Volatile facts that auto-expire after about 12 hours. Good for
tracking transient details (current task, what someone just said,
today's weather) that shouldn't become permanent memories.

    # Store a fact
    mem.add_fact(entity="Alice", attribute="favorite_drink", value="oat latte")
    mem.add_fact(entity="Alice", attribute="current_topic", value="weekend plans")

    # Retrieve facts for an entity
    facts = mem.get_facts(entity="Alice")
    for f in facts:
        print(f"{f.attribute}: {f.value}")

    # Or get a formatted context string
    stm_ctx = mem.get_stm_context(entity="Alice")

    # Context-based retrieval (keyword matching)
    relevant_facts = mem.get_facts(context="coffee preferences")

STM auto-deduplicates: updating the same entity+attribute replaces
the old fact rather than stacking duplicates.


ENTITY PREFERENCES
------------------
Track structured likes, dislikes, and sentiments about topics for
each entity. More structured than free-text social impressions.

    mem.update_entity_preference("Alice", "music", "jazz", "likes")
    mem.update_entity_preference("Alice", "music", "heavy metal", "dislikes")
    mem.update_entity_preference("Alice", "food", "sushi", "loves")

    # Get all preferences
    prefs = mem.get_entity_preferences("Alice")
    # {"music": {"jazz": "likes", "heavy metal": "dislikes"}, "food": {"sushi": "loves"}}

    # Get formatted context string (used in get_context_block() automatically)
    pref_ctx = mem.get_preference_context("Alice")


PROFESSIONAL MODE
-----------------
For task-oriented agents (coding assistants, project managers, etc.)
that don't need emotional coloring. Professional mode:

  - Freezes mood at neutral — no emotional drift
  - Disables mood-congruent recall bias
  - Uses neutral context block headers ("=== MEMORY ===" instead of
    "=== WHAT I REMEMBER ===")
  - Preserves ALL core mechanics: decay, spaced repetition, resonance,
    deduplication, task memory, solutions, artifacts

    mem = VividnessMem(professional=True)

    # Everything works the same, minus emotion effects
    mem.add_self_reflection("The API uses pagination with cursor tokens",
                           emotion="neutral", importance=7)
    mem.update_mood_from_conversation(["frustrated"])  # no-op
    print(mem.mood_label)  # always "neutral"


TASK MEMORY (Project / Task / Solution / Artifact tracking)
-----------------------------------------------------------
Full project-aware task memory for agents that build things. Tracks
projects, tasks, actions, auto-extracted solutions, and artifacts.

### Projects

    mem.set_active_project("my-web-app")
    projects = mem.list_projects()   # [{"name": "my-web-app", "status": "active", ...}]
    mem.archive_project("old-project")

### Tasks

    # Start a task (returns a unique task_id)
    tid = mem.start_task("Implement user authentication",
                         project="my-web-app", priority=8,
                         tags=["backend", "security"])

    # Subtasks via parent_id
    sub_tid = mem.start_task("Write JWT token validator",
                             project="my-web-app", parent_id=tid)

    # Complete or fail
    mem.complete_task(tid, outcome="Auth system working with JWT + refresh tokens")
    mem.fail_task(sub_tid, reason="Token library has a breaking bug")
    # Also: mem.abandon_task(tid, reason="Requirements changed")

    # Query tasks
    active = mem.get_active_tasks()                 # current project
    active = mem.get_active_tasks("my-web-app")     # specific project
    task = mem.get_task(tid)                         # by ID

### Action Logging

    # Log what the agent did while working on a task
    mem.log_action(tid, action="Added bcrypt password hashing",
                   result="success", importance=6)

    mem.log_action(tid, action="Tried MD5 for passwords",
                   result="failure", error="MD5 is insecure",
                   fix="Switched to bcrypt", importance=7)

    actions = mem.get_task_actions(tid)

When the system detects a failure followed by a success on the same
task, it automatically extracts a SolutionPattern — "I tried X, it
failed, then I did Y and it worked."

### Solution Patterns

    # Record a solution explicitly
    mem.record_solution(
        problem="CORS errors when calling API from frontend",
        failed_approaches=["Adding headers manually", "Using a proxy"],
        solution="Configure CORS middleware on the server with allowed origins",
        tags=["backend", "cors"],
        importance=8,
    )

    # Find matching solutions later
    matches = mem.find_solutions("getting CORS errors on fetch requests")
    for sol in matches:
        print(f"Solution: {sol.solution}")
        print(f"Failed approaches: {sol.failed_approaches}")
        print(f"Applied {sol.times_applied} times")

    # Solutions gain importance each time they're applied (capped at +3)

### Artifact Tracking

    # Track key project elements — files, APIs, characters, concepts
    mem.track_artifact("auth_middleware.py", artifact_type="file",
                       description="JWT authentication middleware",
                       project="my-web-app", importance=8,
                       dependencies=["jwt_utils.py"])

    mem.update_artifact("auth_middleware.py",
                        state="refactored to use refresh tokens")

    # Get a compact overview of all project artifacts
    overview = mem.get_project_overview("my-web-app")

    # Find related artifacts (dependency graph)
    related = mem.get_related("auth_middleware.py")

### Task Context Injection

    # Build the task memory section for your system prompt
    task_ctx = mem.get_task_context(conversation_context="working on auth")
    # Includes: active project, tasks, recent solutions, artifacts

    # Or let get_context_block() include it automatically:
    full_ctx = mem.get_context_block(
        current_entity="User",
        conversation_context="working on auth",
    )

### Task Memory Decay

Task memory uses tiered decay based on project age:
  - Active (0–7 days): Normal decay — everything fresh
  - Cooling (8–30 days): 2.5× faster decay — details fade
  - Cold (31–90 days): 6× faster decay — only key solutions persist
  - Archived: No tasks shown, but solutions remain searchable


SESSION MANAGEMENT & MAINTENANCE
---------------------------------
Call these between sessions:

    # Bump session counter
    session_num = mem.bump_session()

    # Check if it's time for maintenance
    if mem.needs_brief():
        prompt = mem.prepare_brief_prompt(entity="Alice")
        response = your_llm.generate(prompt)
        parsed = parse_brief_response(response)
        mem.apply_brief(parsed, entity="Alice")

    if mem.needs_rescore():
        prompt, indexed = mem.prepare_rescore_prompt()
        if prompt:
            response = your_llm.generate(prompt)
            adjustments = parse_rescore_response(response)
            mem.apply_rescores(adjustments, indexed)

    # Save everything
    mem.save()
    mem._save_brief()


MEMORY CONSOLIDATION
--------------------
Periodically (e.g. every 10 sessions), consolidate related memories:

    # Find clusters of related memories
    clusters = mem.find_consolidation_clusters()

    # Or use the combined prompt builder
    prompt = mem.prepare_consolidation_prompt()
    if prompt:
        response = your_llm.generate(prompt)
        gists = parse_curation_response(response)  # same JSON format
        mem.apply_consolidation(gists)
        mem.save()


CONTRADICTION DETECTION
-----------------------
The system automatically flags contradictions in get_context_block().
You can also check manually:

    pairs = mem.detect_contradictions()
    for older, newer in pairs:
        print(f"Conflict: '{older.content}' vs '{newer.content}'")

    # With LLM verification for catching subtle conflicts:
    pairs = mem.detect_contradictions(llm_fn=your_llm.generate)

    # Get formatted contradiction context
    ctx = mem.get_contradiction_context()


ASSOCIATION CHAINS
------------------
Walk the association graph from seed memories to find conceptually
connected memories that didn't directly match a query:

    active = mem.get_active_self()
    seed_indices = [0, 2]  # indices into self_reflections
    associated = mem.associate(seed_indices, hops=2)

FILE STRUCTURE
--------------
VividnessMem persists everything as plain JSON:

    memory_data/
      self_memory.json      # all self-reflections
      brief.json            # compressed briefs, session counter, mood
      task_memory.json      # projects, tasks, actions, solutions, artifacts
      social/
        alice.json           # impressions of Alice
        bob.json             # impressions of Bob


ENCRYPTION (optional)
---------------------
VividnessMem supports opt-in at-rest encryption for all stored data.
When enabled, every JSON file on disk is encrypted with AES-128-CBC +
HMAC-SHA256 (Fernet), and entity filenames are hashed with HMAC-SHA256
so real names never appear on the filesystem.

Requires the `cryptography` package:

    pip install cryptography

Enable it by passing a password string to the constructor:

    mem = VividnessMem(encryption_key="my-secret-passphrase")

That's the only change — all other API calls remain identical.

    # Same API, encrypted on disk
    mem.add_self_reflection(content="I like jazz", emotion="happy", importance=7)
    mem.save()

    # To load later, pass the same key
    mem2 = VividnessMem(encryption_key="my-secret-passphrase")
    print(mem2.self_reflections)  # decrypted in memory

    # Wrong password = graceful fallback (empty memory, logged warning)
    mem3 = VividnessMem(encryption_key="wrong-password")
    print(mem3.self_reflections)  # []

How it works:
  - Password is derived into a Fernet key via PBKDF2-HMAC-SHA256
    with 480,000 iterations and a random 16-byte salt.
  - The salt is stored once in <data_dir>/.salt (created on first use).
  - Encrypted files use the .enc extension instead of .json.
  - Entity filenames become HMAC-SHA256 hashes (e.g. a3f1b2c9e7d0.enc).
  - If encryption_key is None (default), everything stays plaintext —
    no extra dependencies needed.

Encrypted file structure:

    memory_data/
      .salt                  # 16-byte random salt (keep this safe!)
      self_memory.enc        # encrypted self-reflections
      brief.enc              # encrypted briefs, session counter, mood
      task_memory.enc        # encrypted task memory
      social/
        a3f1b2c9e7d04f1a.enc # encrypted impressions (hashed name)
        7bc0de913fa48e22.enc # another entity

Security notes:
  - The .salt file is critical — if lost, data cannot be decrypted.
  - Store the encryption key securely (env variable, keyring, etc.).
  - This protects data at rest. In-memory data is always plaintext.
  - You cannot mix encrypted and plaintext modes on the same data_dir.


MEMORY DREAMING
---------------
Between sessions, VividnessMem can "dream" — replaying memories against
each other to find hidden connections that were never in the same context
window. It uses the association graph to find 2-hop pairs (memories
connected through an intermediary but never directly linked).

    from VividnessMem import VividnessMem, DREAM_PROMPT, parse_dream_response

    mem = VividnessMem()
    session = mem.bump_session()

    # Check if it's time to dream (every DREAM_INTERVAL sessions, default 2)
    if mem.needs_dream():
        prompt = mem.prepare_dream_prompt()
        if prompt:
            response = your_llm.generate(prompt)
            connections = parse_dream_response(response)
            mem.apply_dream(connections)
            mem.save()

How it works:
  - find_dream_candidates() walks the association graph 2 hops deep
    to find memory pairs that share a bridge concept but were never
    directly co-active.
  - The LLM receives these pairs and is asked to find hidden connections.
  - apply_dream() stores each new insight as a memory with source="dream".
  - Duplicate insights are automatically filtered out.
  - Dream log is stored in brief data for tracking.


REGRET SCORING
--------------
When the periodic rescore lowers a memory's importance, VividnessMem
tracks the "regret" — how much the agent overestimated that memory.
This builds a picture of what kinds of things the agent tends to
overrate, enabling metacognitive self-awareness.

Regret is tracked automatically during apply_rescores() — no extra
API calls needed. The regret context appears in get_context_block()
once 3+ memories have been flagged.

    # Check regret patterns manually
    patterns = mem.get_regret_patterns()
    print(patterns)
    # {
    #   "count": 4,
    #   "avg_original_importance": 8.5,
    #   "avg_final_importance": 6.0,
    #   "common_emotions": ["excitement", "surprise"]
    # }

    # Get the regret context string (empty if < 3 regretted memories)
    ctx = mem.get_regret_context()
    # "=== THINGS I'VE LEARNED ABOUT MY OWN JUDGMENT ===\n..."

    # Inspect individual regretted memories
    for m in mem.get_regret_memories():
        print(f"regret={m._regret:.2f}  believed={m._believed_importance}  now={m.importance}")

How it works:
  - When apply_rescores() drops importance (with original >= 5),
    the memory gets _regret = (original - clamped) / 10.0
  - _believed_importance preserves the original before correction.
  - Regret values range from 0.0 to ~0.2 (capped by ±2 adjustment limit).
  - The regret context section only appears after 3+ flagged memories,
    preventing noise from isolated corrections.


RELATIONSHIP ARC TRACKING
--------------------------
VividnessMem tracks per-entity trajectories — not just static social
impressions but whether each relationship is warming or cooling over
time. Each add_social_impression() call automatically updates the arc.

    # Arcs update automatically when you add social impressions
    mem.add_social_impression(
        entity="Alice",
        content="Alice challenged my ideas today — tough but fair",
        emotion="respect",
        importance=7,
    )

    # Get the arc for an entity
    arc = mem.get_relationship_arc("Alice")
    print(arc)
    # {
    #   "trajectory": 0.23,        # EMA of emotion valence (-1 to +1)
    #   "warmth": 0.15,            # accumulated warmth (-1 to +1)
    #   "history": [[1, 0.05], [2, 0.15]],  # [session, warmth] over time
    #   "impression_count": 5,
    #   "trend_label": "warming"   # warming / cooling / stable
    # }

    # Get a one-line arc description (used in context block automatically)
    print(mem.get_arc_context("Alice"))
    # "(My relationship with Alice feels warm and warming. Trajectory: +0.23)"

How it works:
  - trajectory is an exponential moving average (alpha=0.35) of
    emotion valence from each social impression.
  - warmth accumulates slowly based on trajectory direction,
    modeling how relationships build or erode over time.
  - Trend labels: trajectory > 0.15 = "warming", < -0.15 = "cooling",
    otherwise "stable".
  - Arc context appears automatically in get_context_block() when
    the entity has 2+ impressions.


KEY CONCEPTS
------------
- Vividness:    importance * retention_curve(age, stability)
                Memories naturally fade. Accessing them at the right
                time reinforces them (spaced repetition).

- Mood:         PAD vector (Pleasure, Arousal, Dominance).
                Shifts gradually with conversation emotions.
                Biases which memories surface (mood-congruent recall).
                Disabled in professional mode.

- Resonance:    Old faded memories resurfacing when the current
                conversation matches their keywords. Uses an inverted
                index with bigram/trigram matching for O(k) lookup.

- Associations: Memories linked by shared concepts. When one resonates,
                connected memories can chain-activate.

- Foreground/   Active memories split by relevance to the current
  Background:   conversation. Foreground gets full text; background
                gets compressed one-liners. Saves prompt tokens.

- Touch         Memories only get reinforced if they share keywords
  Dampener:     with the current context. Prevents blind top-N boosting.

- Deduplication: New memories with >80% word overlap merge with existing
                 ones (keeping higher importance, preserving history).

- STM:          Short-term volatile facts (12-hour decay) for transient
                details. Auto-deduplicates by entity+attribute.

- Preferences:  Structured entity likes/dislikes tracked separately
                from free-text social impressions.

- Briefs:       Periodic LLM-generated compressed self-summaries that
                capture patterns across many memories.

- Rescoring:    Periodic LLM re-evaluation of importance scores based
                on actual access patterns. Includes emotional reappraisal.

- Consolidation: Clustering related memories and asking the LLM to
                 synthesize them into gist insights.

- Contradictions: Automatic detection of memories that may conflict
                  (same topic, opposing emotion or negation patterns).
                  Optional LLM verification for subtle conflicts.

- Dreaming:     Between-session discovery of hidden connections via
                2-hop association graph traversal. Finds memories that
                were never co-active but share latent links.

- Regret:       Metacognitive tracking of overestimated importance.
                When rescoring drops a memory's importance, the original
                belief is preserved and a regret score is assigned.

- Relationship  Per-entity trajectory tracking. EMA of emotion valence
  Arcs:         (trajectory) plus accumulated warmth. Automatically
                labels warming/cooling/stable trends.

- Task Memory:  Project-aware task/action/solution/artifact tracking
                with tiered decay (active → cooling → cold → archived).
                Solutions auto-extracted from fail→success sequences.

- Professional  Emotion-free mode for task-oriented agents. All core
  Mode:         mechanics preserved (decay, resonance, dedup, tasks),
                only mood and emotional biasing disabled.


TUNING
------
Override class constants to adjust behavior:

    mem = VividnessMem()
    mem.ACTIVE_SELF_LIMIT = 12       # surface more memories per turn
    mem.RESONANCE_LIMIT = 5          # allow more resonant memories
    mem.RESONANCE_SCORE_FLOOR = 2.0  # lower = more lenient resonance


FULL API REFERENCE (quick list)
-------------------------------
Core:           VividnessMem(data_dir, encryption_key, professional)
                .save()  .stats()  .bump_session()

Self Memory:    .add_self_reflection(content, emotion, importance, source, why_saved)
                .get_active_self(context)  .partition_active_self(context)

Social:         .add_social_impression(entity, content, emotion, importance, why_saved)
                .get_active_social(entity)

Mood:           .mood  .mood_label  .update_mood_from_conversation(emotions)

STM:            .add_fact(entity, attribute, value, retrieval_tags)
                .get_facts(entity, context)  .get_stm_context(entity, context)

Preferences:    .update_entity_preference(entity, category, item, sentiment)
                .get_entity_preferences(entity)  .get_preference_context(entity)

Resonance:      .resonate(context, limit, llm_fn)
Association:    .associate(seed_indices, hops)

Tasks:          .set_active_project(name)  .list_projects()  .archive_project(name)
                .start_task(desc, project, priority, parent_id, tags)
                .complete_task(id, outcome)  .fail_task(id, reason)
                .abandon_task(id, reason)
                .get_active_tasks(project)  .get_task(id)

Actions:        .log_action(task_id, action, result, error, fix, importance)
                .get_task_actions(task_id)

Solutions:      .record_solution(problem, failed, solution, tags, importance)
                .find_solutions(description, top_k)

Artifacts:      .track_artifact(name, type, desc, project, importance, deps)
                .update_artifact(name, state, desc)
                .get_project_overview(project)  .get_related(name)

Context:        .get_context_block(entity, resonant, conversation_context)
                .get_task_context(conversation_context)
                .get_contradiction_context()  .get_regret_context()
                .get_arc_context(entity)  .get_preference_context(entity)
                .get_stm_context(entity, context)

Maintenance:    .needs_brief()  .prepare_brief_prompt(entity)
                .apply_brief(parsed, entity)
                .needs_rescore()  .prepare_rescore_prompt()
                .apply_rescores(adjustments, indexed)
                .detect_contradictions(limit, llm_fn)
                .find_consolidation_clusters()
                .prepare_consolidation_prompt()  .apply_consolidation(gists)
                .needs_dream()  .prepare_dream_prompt()  .apply_dream(connections)
                .get_regret_memories()  .get_regret_patterns()
                .get_relationship_arc(entity)

Parsing:        parse_curation_response(text)  parse_brief_response(text)
                parse_rescore_response(text)  parse_dream_response(text)
                expand_query(llm_fn, query)
